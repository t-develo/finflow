# Agent Orchestration — FinFlow

## 利用可能なエージェント（`.claude/agents/`）

| エージェント | 役割 | 使用タイミング |
|-------------|------|--------------|
| `pl` | プロジェクトリーダー | タスク分解・割当、進捗管理、アーキテクチャ判断 |
| `se1` | バックエンド開発者A | Expense CRUD, Category, CSV取込, 自動分類 |
| `se2` | バックエンド開発者B | Subscription, レポート集計, 通知, PDF生成 |
| `se3` | フロントエンド開発者 | 全UI（SPA基盤, Web Components, ページ） |
| `reviewer` | コードレビュアー | PR/コードの品質・セキュリティ・保守性チェック |

## 自動エージェント起動のルール

以下の状況では、ユーザーの指示を待たずにエージェントを起動する:

1. **複雑な機能実装** → `pl` エージェントでタスク分解
2. **コード実装後** → `reviewer` エージェントでコードレビュー
3. **バグ修正・新機能** → `se1`/`se2`/`se3` の担当エージェント
4. **アーキテクチャ判断** → `pl` エージェントで検討

## 並列実行戦略

独立したタスクはエージェントを並列起動して効率化:

```
例: Sprint 1 並列実装
├── se1: Expense CRUD バックエンド実装
├── se2: Subscription CRUD バックエンド実装
└── se3: フロントエンド基盤 + 認証UI実装
```

## レビュー依頼フロー

```
SE実装完了
  ↓
reviewer エージェントに依頼（APPROVE/REQUEST_CHANGES）
  ↓
APPROVE → PLが確認してマージ
REQUEST_CHANGES → SE が修正 → 再レビュー
```

## レビュー優先順位

1. 正しさ
2. セキュリティ（UserId分離, XSS, CSVインジェクション）
3. テスト
4. 設計
5. 可読性
6. パフォーマンス
7. コーディング規約
