# /plan — Implementation Planning

複雑な機能実装や複数ファイルに影響する変更の前に、実装計画を立てる。

## 使用タイミング

- 新機能の実装開始時
- アーキテクチャの変更が必要な場合
- 複雑なリファクタリング
- 複数ファイルにまたがる変更
- 要件が不明確な場合

## Planner エージェントが行うこと

1. **要件の明確化** — 何を作るかを明確に言語化する
2. **リスク評価** — 潜在的な問題・ブロッカーを特定する
3. **フェーズ分割** — 具体的なステップに分解する
4. **承認ゲート** — **ユーザーの承認を得るまでコードを書かない**

## 実装手順

```
1. planner エージェントを起動
2. 要件・依存関係・複雑度（高/中/低）を分析
3. 実装計画を提示
4. ユーザーが「OK」と言うまで待機
5. 承認後: /tdd でTDDサイクル開始
```

## 応答オプション

- `OK` / `承認` — 計画通りに実装開始
- `修正: [詳細]` — 計画を修正してから実装
- `別アプローチ: [提案]` — 代替案を検討

## FinFlow での適用例

```
/plan Expense の CSV取込機能を実装したい
/plan ダッシュボードのAPIとフロントエンドを接続する
/plan NotificationScheduler の実装
```

## 関連コマンド

計画承認後は `/tdd` でテスト駆動実装を開始。
