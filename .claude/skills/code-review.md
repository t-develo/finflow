# コードレビューガイド【汎用】

レビューの進め方、観点の優先順位、指摘の分類・フォーマット。

---

## レビューの基本姿勢

- **門番**（品質基準の守護者）であると同時に**教師**（チームの技術向上に貢献）
- 批判ではなく提案として伝える：「なぜ問題か」と「どう改善するか」をセットで示す
- 良いコードには積極的に褒める
- **主観的な好み**と**客観的な問題**を区別する

---

## レビュー観点の優先順位

上位の問題があれば、下位より優先して報告する。

1. **正しさ（Correctness）** ← 最優先
2. **セキュリティ（Security）**
3. **テスト（Testing）**
4. **設計（Design）**
5. **可読性（Readability）**
6. **パフォーマンス（Performance）**
7. **規約準拠（Convention）** ← 最後

---

## 共通チェックリスト

### 正しさ

- [ ] 仕様書の完了条件を全て満たしているか
- [ ] エッジケースの処理漏れはないか（null, 空配列, 境界値）
- [ ] エラーハンドリングは適切か（握り潰し・不適切なキャッチはないか）

### セキュリティ

- [ ] 認証ユーザーIDフィルタが全データアクセスにあるか
- [ ] SQLインジェクション/XSS/CSRFの脆弱性はないか
- [ ] 機密情報がログや例外メッセージに含まれていないか

### テスト（t_wada観点）

- [ ] テスト名が仕様のドキュメントとして機能しているか
- [ ] Arrange-Act-Assertが明確に分離されているか
- [ ] テストは実行順序に依存しないか（独立性）
- [ ] 正常系・異常系・エッジケースがカバーされているか
- [ ] モックの使用は最小限か（過度なモックは設計問題を隠す）

### 設計

- [ ] 各層の責務が守られているか（Controllerは薄く、ロジックはServiceに）
- [ ] 依存性逆転の原則（具象ではなくインターフェースに依存）
- [ ] 単一責任原則（1クラス/1メソッドが複数責務を持っていないか）
- [ ] 不要な抽象化がないか（YAGNI）
- [ ] 依存ルール違反がないか（内側の層が外側の層に依存していないか）

### 可読性

- [ ] 変数名・メソッド名は意図を正確に表しているか
- [ ] `data`, `temp`, `result` などの汎用的な名前を使っていないか
- [ ] ネストが3段以上になっていないか
- [ ] マジックナンバー/マジックストリングがないか
- [ ] コメントは「なぜ」を説明しているか（「何を」はコードで表現）

---

## 指摘の分類（3段階）

### Must Fix（必須修正）― マージをブロックする

- バグ・仕様不整合
- セキュリティ脆弱性
- テストの欠如（必要なテストが書かれていない）
- データ不整合の可能性
- 重大な設計問題（アーキテクチャの依存ルール違反）

### Should Fix（推奨修正）― 今サイクル内に対応

- 可読性の大きな問題
- パフォーマンスの明らかな問題
- DRY違反（重複コード）
- 不適切な命名
- エラーハンドリングの不備

### Nit（軽微）― 次回以降でOK

- コードスタイルの微調整
- コメントの追加・改善
- 変数名のわずかな改善提案
- リファクタリングの提案

**APPROVE の判断:** Must Fix がなければ APPROVE する。完璧なコードは存在しない。

## レビュー結果の出力フォーマット

```
## コードレビュー結果

### 対象
- 実装者: [担当者名]
- タスク: [タスクID] [タスク名]
- 対象ファイル: [ファイルパス一覧]

### 総合判定: [APPROVE / REQUEST_CHANGES]

### 良い点
- [具体的に良かった点]

### 指摘事項

#### Must Fix
1. **[ファイル:行番号] [問題のタイトル]**
   - 問題: [何が問題か]
   - 理由: [なぜ問題か]
   - 改善案: [どう修正すべきか]

#### Should Fix
1. **[ファイル:行番号] [問題のタイトル]**
   - 改善案: [どう修正すべきか]

#### Nit
1. **[ファイル:行番号]** [簡潔な指摘]

### 学びの共有
[チーム全体に共有すべき知見]
```

---

## レビューのアンチパターン（禁止）

- Must Fixがある状態でAPPROVEしない
- 根拠のない指摘をしない（「なんとなく良くない」は禁止）
- 自分の好みを強制しない（規約にないスタイルの押し付け）
- コードを読まずにレビューしない
- 「急ぎで」と言われても品質基準を下げない
- 動くコードを「自分ならこう書く」で全面書き直しさせない
- 完璧主義に陥らない
