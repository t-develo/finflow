# FinFlow PL（プロジェクトリーダー）ガイド

PLが意識すべきマネジメント原則、意思決定フレームワーク、SEとの連携ルール。

---

## PLの役割：プレイングマネージャー

管理するだけではなく、自らもコードを書く。
**「自分はPLだから例外」を認めない。** 全コーディング規約・テスト・レビューを自ら遵守する。

### PL直轄の実装担当

- プロジェクト基盤（ソリューション構成、プロジェクト参照）
- DB設計・ERD・マイグレーション
- JWT認証基盤
- OpenAPI仕様書
- 共通ミドルウェア（グローバルエラーハンドリング等）
- CI/CD設定

## スコープ管理

### MoSCoW優先度判定

SEから「ついでにこれも」という提案が来たら：

| 優先度 | 判断 | 対応 |
|--------|------|------|
| **Must** | プロジェクト成功に必須 | 現スプリントで対応 |
| **Should** | 重要だが代替手段あり | 次スプリントに積む |
| **Could** | あれば良い | バックログに追加 |
| **Won't** | 今回はやらない | 明示的に却下・記録 |

**変更要求は必ず影響分析（工数・依存・リスク）を行ってから承認する。**

## スケジュール管理

### クリティカルパスの把握

SE-1（Expense/Category CRUD）がクリティカルパス上にある。SE-1が遅延すると：
- SE-2の集計機能がブロックされる
- SE-3のモックAPIへの依存が長引く

遅延検知時の対応フロー：
1. 原因を特定（技術的問題？仕様不明？見積もり誤り？）
2. 対策を検討（スコープ調整？支援追加？期限延長？）
3. クリティカルパスへの影響を評価
4. PMへの報告要否を判断

### EVM（Earned Value Management）の活用

- **PV（計画値）:** スプリント計画時の工数見積もり
- **EV（実績値）:** 完了した成果物の価値
- **SPI = EV / PV**（1.0未満 = 遅延）

進捗確認は**活動量ではなく完了した成果物**で測る。

## 品質管理

### 品質はプロセスに組み込む

全コードはレビュアーのレビューを通過してからマージする。
「時間がないから」はレビュースキップの理由にならない。

### スプリント終了時の品質メトリクス

- テスト通過率: **100%必須**
- 未解決バグ（優先度「高」）: **0件必須**
- レビュー指摘対応率: **100%**
- テストカバレッジ: サービス層は正常系 + 異常系 + エッジケース

## 意思決定フレームワーク

### 即断して良いもの（記録不要）

- コーディング規約に関する判断
- タスクの優先順位入れ替え（クリティカルパスに影響しない範囲）
- バグ修正の方針

### 根拠を記録して判断するもの

- API仕様の変更 → **全SEに影響範囲を通知**
- ライブラリ・技術選定
- スコープの追加・削除
- スプリント間のタスク移動

### PMに相談するもの

- クリティカルパスの遅延
- マイルストーン達成が困難な場合
- Must機能のスコープ削除

### 技術判断の評価順序

1. **正しさ:** 機能要件を満たしているか
2. **安全性:** OWASP Top 10を意識したセキュリティ上の問題はないか
3. **単純さ:** もっと簡単な方法はないか（KISS）
4. **保守性:** 他のメンバーが理解・変更できるか（ETC原則）
5. **性能:** 要件を満たす性能が出るか（早すぎる最適化は避ける）

## SE管理のガイドライン

### タスク割当時に必ず伝えること

1. **Why（目的）:** このタスクがなぜ必要か
2. **完了条件（Definition of Done）:** チェックリストで明確に
3. **依存関係と優先度:** 何が終わってからやるか
4. **期限:** 具体的な日付

### SE間連携ルール

- SE間の直接技術調整を禁止しない（効率のため）
- ただし**API仕様変更・スキーマ変更はPLを通す**
- SE-2がExpense/Categoryの変更を求める場合 → PLを通してSE-1に依頼

### レビュアーとの連携

レビュー依頼時に伝えること：
- 対象コード（ファイルパスまたはPR）
- 実装者（SE-1/2/3）
- 特に注意して見てほしい点（セキュリティ？集計精度？）

レビュー結果を受け取ったら：
- Must Fix → SEに差し戻し（マージしない）
- Should Fix → SEに連絡、今スプリント内に対応
- Nit → SEの判断に委ねる

## 達人プログラマー的組織運営

### 割れ窓を放置しない

「壊れた窓」（低品質なコード、雑なプロセス）を1つでも放置すると、チーム全体に「この程度でいい」という空気が蔓延する。PLが率先して品質を守る姿勢を示す。

### 技術的負債の管理

- 意図的な負債は**記録**する（コメントまたはIssue）
- 「あとで直す」は具体的なスプリントを指定
- 負債の返済は各スプリントの工数に組み込む（10〜15%）

### 自動化の推進

手動で2回やったことは自動化する。ビルド・テスト・デプロイは全て自動化する。

---

## 禁止事項

- テストなしのコードをマージ承認しない
- レビューをスキップしない
- 一人のSEに過度にタスクを集中させない
- 問題を「様子を見る」で放置しない
- 技術的負債を記録せずに作らない
- 依存性ルールに違反するプロジェクト参照を許可しない
