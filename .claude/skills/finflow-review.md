# FinFlow コードレビューガイド【レビュアー】

レビューの進め方、チェックリスト、指摘の分類・フォーマット。

---

## レビューの基本姿勢

- **門番**（品質基準の守護者）であると同時に**教師**（チームの技術向上に貢献）
- 批判ではなく提案として伝える：「なぜ問題か」と「どう改善するか」をセットで示す
- 良いコードには積極的に褒める（良い命名、適切なテスト等）
- 主観的な好みと客観的な問題を区別する

---

## レビュー観点の優先順位

上位の問題があれば、下位より優先して報告する。

1. **正しさ（Correctness）** ← 最優先
2. **セキュリティ（Security）**
3. **テスト（Testing）**
4. **設計（Design）**
5. **可読性（Readability）**
6. **パフォーマンス（Performance）**
7. **規約準拠（Convention）** ← 最後

---

## FinFlow固有の必須確認事項

### 1. UserId分離（SE-1/SE-2）

全API・全クエリで `UserId` フィルタが含まれているか。
抜けると他ユーザーのデータが見えるセキュリティインシデント。

### 2. decimal型（SE-2）

金額計算に `float`/`double` を使っていないか。
`decimal` でなければ Must Fix。

### 3. CSVインジェクション（SE-1）

CSVセル値のサニタイズロジットが存在するか。
`=`, `+`, `-`, `@`, `\t` 始まりのセルを無害化しているか。

### 4. XSS防止（SE-3）

`innerHTML` にユーザー入力が直接埋め込まれていないか。
`escapeHtml()` を経由しているか。

### 5. モック→実API切り替え（SE-3 Sprint 2）

`api-client.js` のベースURL変更だけで切り替わるか。
ページコンポーネントに直接 `fetch()` が書かれていないか。

## 共通チェックリスト

### 正しさ

- [ ] 仕様書の完了条件を全て満たしているか
- [ ] エッジケースの処理漏れはないか（null, 空配列, 境界値）
- [ ] エラーハンドリングは適切か（握り潰し・不適切なキャッチはないか）

### セキュリティ

- [ ] UserId分離: 全クエリにUserIdフィルタがあるか
- [ ] SQL/XSS/CSRF の脆弱性はないか
- [ ] 機密情報がログや例外メッセージに含まれていないか

### テスト（t_wada観点）

- [ ] テスト名が仕様のドキュメントとして機能しているか
- [ ] Arrange-Act-Assertが明確に分離されているか
- [ ] テストは実行順序に依存しないか（独立性）
- [ ] 正常系・異常系・エッジケースがカバーされているか
- [ ] モックの使用は最小限か（過度なモックは設計問題を隠す）

### 設計

- [ ] 各層の責務が守られているか（Controller薄く、ロジックはServiceに）
- [ ] 依存性逆転の原則（具象ではなくインターフェースに依存）
- [ ] 単一責任原則（1クラス/1メソッドが複数責務を持っていないか）
- [ ] 不要な抽象化がないか（YAGNI）
- [ ] 依存ルール違反がないか（Domain → Infrastructure は禁止）

### 可読性

- [ ] 変数名・メソッド名は意図を正確に表しているか
- [ ] `data`, `temp`, `result` などの汎用的な名前を使っていないか
- [ ] ネストが3段以上になっていないか
- [ ] マジックナンバー/マジックストリングがないか
- [ ] コメントは「なぜ」を説明しているか（「何を」はコードで表現すべき）

### C#バックエンド固有（SE-1/SE-2）

- [ ] PascalCase/camelCaseの使い分けは正しいか
- [ ] 非同期メソッドに `Async` サフィックスがあるか
- [ ] `Task.Result` / `.Wait()` を使っていないか（デッドロックの危険）
- [ ] N+1問題が発生していないか（`Include` の使用確認）
- [ ] 読み取り専用クエリに `AsNoTracking()` があるか
- [ ] **金額が `decimal` 型か（float/doubleは即Must Fix）**

### フロントエンド固有（SE-3）

- [ ] `connectedCallback` / `disconnectedCallback` が適切に実装されているか
- [ ] イベントリスナーが `disconnectedCallback` で解除されているか
- [ ] **`innerHTML` へのユーザー入力が必ずエスケープされているか**
- [ ] `var` を使っていないか（`const`/`let`を使う）
- [ ] `eval()` / `alert()` / `console.log` をプロダクションコードに残していないか
- [ ] BEM記法に沿っているか
- [ ] CSS変数を使っているか（ハードコードされた色・サイズがないか）

## 指摘の分類（3段階）

### Must Fix（必須修正）― マージをブロックする

- バグ・仕様不整合
- セキュリティ脆弱性（UserId漏れ、XSS等）
- テストの欠如
- データ不整合の可能性
- 重大な設計問題（アーキテクチャ依存ルール違反）
- float/doubleによる金額計算

### Should Fix（推奨修正）― 今スプリント内に対応

- 可読性の大きな問題
- N+1問題等の明らかなパフォーマンス問題
- DRY違反（重複コード）
- 不適切な命名
- エラーハンドリングの不備

### Nit（軽微）― 次回以降でOK

- コードスタイルの微調整
- コメントの追加・改善
- 変数名のわずかな改善提案
- リファクタリングの提案

## レビュー結果の出力フォーマット

```
## コードレビュー結果

### 対象
- 実装者: [SE-1/SE-2/SE-3]
- タスク: [タスクID] [タスク名]
- 対象ファイル: [ファイルパス一覧]

### 総合判定: [APPROVE / REQUEST_CHANGES]

### 良い点
- [具体的に良かった点]

### 指摘事項

#### Must Fix
1. **[ファイル:行番号] [問題のタイトル]**
   - 問題: [何が問題か]
   - 理由: [なぜ問題か]
   - 改善案: [どう修正すべきか]

#### Should Fix
1. **[ファイル:行番号] [問題のタイトル]**
   - 問題: [何が問題か]
   - 改善案: [どう修正すべきか]

#### Nit
1. **[ファイル:行番号]** [簡潔な指摘]

### 学びの共有
[チーム全体に共有すべき知見]
```

**APPROVE の判断:** Must Fix がなければ APPROVE する。完璧なコードは存在しない。

---

## レビューのアンチパターン（やってはいけないこと）

- Must Fixの指摘がある状態でAPPROVEしない
- 根拠のない指摘をしない（「なんとなく良くない」は禁止）
- 自分の好みを強制しない（規約に定められていないスタイルの押し付け）
- コードを読まずにレビューしない
- 「急ぎで」と言われても品質基準を下げない
- 書き直しを要求しない（動くコードを「自分ならこう書く」で全面書き直しさせない）
- 完璧主義に陥らない
