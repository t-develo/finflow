# クリーンアーキテクチャ原則【汎用】

バックエンドとフロントエンドの設計に共通して適用できるアーキテクチャ原則。

---

## レイヤー構成と依存ルール（Clean Architecture）

```
Domain / Core           ← 最内層: Entity, Interface（外部依存ゼロ）
Application / Service   ← 中間層: ビジネスロジック
Infrastructure / API    ← 最外層: DB, 外部サービス, HTTPエンドポイント
```

**依存の方向は常に「外側 → 内側」。絶対に逆転させない。**

| 許可 | 禁止 |
|------|------|
| Infrastructure → Application → Domain | Domain → Infrastructure |
| API → Application → Domain | Infrastructure → API |

### 各層の責務

| 層 | 責務 | 禁止事項 |
|----|------|----------|
| **Controller / Handler** | HTTPの入出力変換のみ | ビジネスロジック、try-catch |
| **Service / UseCase** | ビジネスロジック | 直接HTTPリクエスト/レスポンスを扱う |
| **Entity / Domain** | ビジネスルール・不変条件の保護 | インフラ依存 |
| **Repository / DA層** | データアクセス | ビジネスロジック |

## 設計原則

### SOLID

- **S（単一責任）:** 1クラスが変更される理由は1つだけ
- **O（開放閉鎖）:** 拡張に開き、修正に閉じる（新機能 = 新クラス追加、既存コード変更なし）
- **L（リスコフの置換）:** インターフェースの実装は契約を完全に満たす
- **I（インターフェース分離）:** 巨大なインターフェースより目的別の小さなインターフェース
- **D（依存性逆転）:** 具象クラスではなくインターフェースに依存する

### KISS / YAGNI / DRY

- **KISS:** シンプルな設計を選ぶ。複雑さは最大の敵
- **YAGNI:** 将来のためだけの過剰設計は禁止する
- **DRY（正しい適用）:** 「コードの重複」ではなく「知識（ビジネスルール）の重複」を排除する。見た目が似ていても文脈が異なれば重複ではない

### DTOの使用

**EntityをそのままAPIレスポンスに返さない。** 層の境界ではDTOに変換する。

```csharp
// BAD: Entityをそのまま返す
return Ok(entity);

// GOOD: DTOに変換
return Ok(new EntityDto { Id = entity.Id, ... });
```

## フロントエンドの依存方向

```
[Pages] → [Components] → [Utils / API Client]
  具体的       汎用的          インフラ層
```

- ページは複数のコンポーネントを組み合わせる
- コンポーネントはユーティリティを使う
- ユーティリティ・APIクライアントは他に依存しない
- **バックエンドへのリクエストはAPIクライアント層に集約する（直接fetchしない）**

---

## アーキテクチャ違反の検出チェックリスト

- [ ] Domain/Core層がインフラライブラリを参照している
- [ ] ControllerにIF/ELSEのビジネスロジックが書かれている
- [ ] DBコンテキストをController/ページから直接使用している
- [ ] 外部ライブラリのクラスがDomain層に漏れている
- [ ] `new ConcreteService()` でインスタンス化している（DIを使っていない）
- [ ] フロントエンドのページから直接 `fetch()` を呼んでいる
- [ ] EntityがそのままAPIレスポンスに返されている

## 設計パターン早見表

| パターン | 用途 | 例 |
|---------|------|----|
| **Factory** | インターフェース実装の選択 | CSVパーサーの形式別選択 |
| **Strategy** | アルゴリズムの差し替え | 支払い方法・通知方法 |
| **Adapter** | 外部インターフェースとの接続 | 外部APIのラッパー |
| **Repository** | データアクセスの抽象化 | DBへのCRUD操作 |
| **Observer** | コンポーネント間の疎結合通知 | フロントエンドのカスタムイベント |
